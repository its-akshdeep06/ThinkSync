Overview
The backend is the entire intelligence layer of the platform — it handles authentication, repository
ingestion, knowledge graph construction, vector search, the LangGraph agent, PR generation, and the API
that the frontend consumes. The frontend talks only to the GraphQL API; everything else is internal.
Build in this sequence. Each layer depends on the one above it — do not skip ahead.
Layer 1
GitHub OAuth & App — authentication, repo access, webhook
registration
Layer 2
Repository ingestion pipeline — clone, walk, tree-sitter AST parsing
Layer 3
Neo4j knowledge graph — nodes, edges, impact traversal queries
Layer 4
Vector store — code embeddings, semantic search via ChromaDB
Layer 5
LangGraph intent engine — 6-node stateful agent with cycles
Layer 6
Change planner & PR generator — Git Trees API, structured PR body
Layer 7
GraphQL API, Bull job queue, webhook handler
The Python AI engine (Layers 2–6) and the Node.js API (Layer 7) run as separate services. The Node.js
backend enqueues jobs via Redis; workers pick them up and call the Python engine over HTTP.

Layer 1 — GitHub Authentication & Repository Access
AUTH
What to build
A GitHub App — not a plain OAuth app. The distinction matters for production: a GitHub App authenticates
as its own identity per repository, has fine-grained permissions, and gets 15,000 API requests/hour vs
5,000 for OAuth. For the hackathon MVP, an OAuth app is faster to set up, but plan the migration.
How it works
Register the app on GitHub's developer portal. You receive a private key (PEM file) and an app ID. When
a user installs the app on their repo, GitHub sends an installation event to your webhook endpoint. Your
backend:
 Receives the installation ID from the webhook payload
 Signs a JWT using your private key + app ID (expires in 10 minutes)
 Exchanges the JWT for an installation access token via GitHub API
 Uses that token for all subsequent calls to that repo (refresh before expiry — tokens last 1 hour)
Libraries
@octokit/auth-app
JWT generation and installation token exchange (Node.js)
@octokit/rest
All GitHub API calls — repos, branches, commits, PRs
PyGithub
Python-side GitHub API access in the AI engine pipeline
PostgreSQL tables to create
users (id, github_id, access_token, created_at)
repositories (id, user_id, github_repo_url, installation_id, indexed_at, status)
jobs (id, repo_id, type, status, intent_text, pr_url, outcome, created_at)
■ The GitHub private key must never live in plaintext env vars. Use AWS Secrets Manager or Parameter Store and
fetch at runtime.

Layer 2 — Repository Ingestion Pipeline
INGESTION
This is the most foundational component. The quality of everything downstream — the graph, the vector
store, the agent's context — depends entirely on how accurately and completely this pipeline extracts code
structure.
Step 1 — Clone the repository
Shell out to git clone inside a Docker container. For large repos use a shallow clone (--depth=1) to get only
the latest snapshot. Store files temporarily on disk or stream directly to the parser. After parsing, clean up
the cloned directory.
 Skip: node_modules, __pycache__, .git, dist, build, vendor, .venv
 Keep: .py, .js, .ts, .jsx, .tsx — expand to .java, .go, .rs in Phase 2
Step 2 — Walk the file tree
Recursively walk every file after cloning. For each file record its full path, detected language (by
extension), size in bytes, and last modified timestamp. This manifest is what drives the parser loop and
also what gets stored as File nodes in Neo4j.
Step 3 — Parse each file with tree-sitter
Install py-tree-sitter and the grammars for each target language. For each file, build the AST and walk it to
extract:
 Every function definition — name, start line, end line, parameter list
 Every class definition — name, list of parent classes for INHERITS edges
 Every import statement — what is imported and from where
 Every function call — which function calls which (for CALLS edges)
Cross-file resolution is the critical step. When the parser sees 'from payment.processor import charge', it
must resolve that to the actual charge function in payment/processor.py. Do this in two passes:
 Pass 1: build an import map — file path → list of exported symbols
 Pass 2: resolve all import statements against the map, emit directed IMPORTS edges
from tree_sitter import Language, Parser
Language.build_library('build/langs.so', ['vendor/tree-sitter-python'])
PY = Language('build/langs.so', 'python')
parser = Parser(); parser.set_language(PY)
tree = parser.parse(bytes(source_code, 'utf8'))
Step 4 — Emit entity and edge lists
Output two lists from the parser: nodes (function, class, file, variable) and edges (CALLS, IMPORTS,
INHERITS, DEFINES). These lists are batch-written to Neo4j in Layer 3.
■ Run ingestion as a background Bull job — it can take several minutes on large repos. The frontend polls job
progress via GraphQL subscriptions, not by waiting on the HTTP response.
■ Reference codebase: GitNexus on GitHub has a production TypeScript implementation of this entire pipeline with
14-language support and constructor-inferred type resolution. Study its AST walker before writing your own.

Layer 3 — Neo4j Knowledge Graph
GRAPH DB
Schema design
Run these Cypher constraints once at startup. Use dot-separated fully-qualified IDs for functions — e.g.
payment.processor.charge_card — never just the bare function name, which can collide across files.
CREATE CONSTRAINT IF NOT EXISTS FOR (f:File) REQUIRE f.path IS UNIQUE;
CREATE CONSTRAINT IF NOT EXISTS FOR (fn:Function) REQUIRE fn.id IS UNIQUE;
CREATE CONSTRAINT IF NOT EXISTS FOR (c:Class) REQUIRE c.id IS UNIQUE;
Node types
File
path, language, lastModified, size
Function
id (FQN), name, startLine, endLine, complexity, language
Class
id (FQN), name, file
Import
module, alias
Package
name, version
Edge types
DEFINES
File → Function, File → Class
CALLS
Function → Function (property: line number)
IMPORTS
File → File or File → Package
INHERITS
Class → Class
BELONGS_TO
Function → Class
Writing data
Use the Neo4j Bolt driver. Always use MERGE (not CREATE) so that re-indexing after a code push
updates properties rather than creating duplicate nodes. Batch writes using UNWIND for performance:
UNWIND $nodes AS n
MERGE (f:Function {id: n.id})
SET f += n.props
The critical impact traversal query
This runs every time a user submits an intent. Given a target function, it finds everything that depends on it
within N hops:
MATCH (target:Function {id: $function_id})
MATCH path = (caller)-[:CALLS*1..5]->(target)
RETURN DISTINCT caller.id, length(path) AS distance
ORDER BY distance

Layer 4 — Vector Store & Semantic Search
VECTOR
The graph handles structural dependencies. The vector store handles semantic similarity. Both are
needed because a developer might say 'update the authentication logic' while the relevant function is
named validateSession — the graph alone won't find it without a starting node, but vector search will.
What to embed
Embed at the function or class level — not entire files. For each function, create a text chunk combining
the function name, its docstring (if present), and its first ~20 lines of body. Store alongside metadata:
{ file_path, function_id, language, node_type, start_line }
MVP setup — ChromaDB + OpenAI
ChromaDB runs in-process with zero infrastructure, persists to disk, and has native LangChain integration.
One collection per repository (named by repo ID):
import chromadb, openai
client = chromadb.Client()
col = client.create_collection('repo-{repo_id}')
emb = openai.embeddings.create(model='text-embedding-3-small',
input=chunk).data[0].embedding
col.add(ids=[chunk_id], embeddings=[emb], metadatas=[meta])
Querying at intent time
q_emb = openai.embeddings.create(model='text-embedding-3-small',
input=intent).data[0].embedding
results = col.query(query_embeddings=[q_emb], n_results=20)
Production options
ChromaDB
MVP — embedded, zero config, Python-native, LangChain support
Qdrant
Self-hosted production — MIT license, fastest ANN, rich filtering
pgvector
If already running Postgres — just CREATE EXTENSION vector, no
extra infra
Pinecone
Managed cloud at scale — serverless, hybrid search, metadata filtering
Hybrid retrieval
At query time, run both a Cypher traversal on Neo4j (structural context) and a vector query on ChromaDB
(semantic context). Merge results, deduplicate by function ID, and rank by combined score. This
dual-retrieval approach consistently outperforms either method alone.
■ Embedding model choices: OpenAI text-embedding-3-small (cloud, 1536 dims, best quality) |
sentence-transformers/all-MiniLM-L6-v2 (local, free, 384 dims) | microsoft/codebert-base (code-specific, excellent
for function-level).

Layer 5 — LangGraph Intent Engine
AGENT
This is the most technically novel component. Build it as a stateful LangGraph graph — not a simple
LangChain chain. The reason is cycles: the validator node must be able to reject a change and send
control back to the planner for a retry. LangChain chains cannot do this; LangGraph can.
The state object
A single state dict flows through all nodes and is updated at each step:
state = {
intent: str,
structured_intent: { target_modules, action_type, constraints, risk },
graph_context: list, # nodes from Neo4j traversal
semantic_context: list, # chunks from ChromaDB
merged_context: str,
change_set: list, # [{file_path, original, new_code, justification}]
validation_errors: list,
retry_count: int
}
Node 1 — intent_parser
Receives the raw natural language string. Calls GPT-4 with a Chain-of-Thought prompt that forces the
model to reason through what modules are involved, what input/output changes are expected, and what
constraints must be preserved. Output is structured JSON.
Prompt: 'Think step by step:
1. What modules/files are DIRECTLY involved?
2. What are the expected INPUT/OUTPUT changes?
3. What constraints must be preserved?
Output JSON: {target_modules, action_type, constraints, estimated_risk}'
Node 2 — graph_retriever
Takes target_modules from the previous node, looks each up in Neo4j to get its node ID, then runs the
K-hop impact traversal (start with K=3). Returns all affected function and class nodes with their source
code.
Node 3 — semantic_search
Takes the original intent string, embeds it, queries ChromaDB for the top 20 semantically similar code
chunks. Catches things the structural traversal missed — especially code with misleading names.
Node 4 — context_builder
Merges graph results and semantic results. Deduplicates by function ID. Builds a single structured context
block injected into the change planner's prompt — includes affected code, dependency map, and user
constraints.


---PAGE---

Node 5 — change_planner
The main GPT-4 call. Receives the structured context and produces a list of atomic changes. Instruct the
model to preserve existing function signatures unless the intent explicitly requires changing them.
Output: [{file_path, function_name, original_code, new_code, justification}]
Node 6 — validator
Re-parses each piece of new_code through tree-sitter to check for syntax errors. Resolves all imports in
the changed code. If any check fails, sends the failed change back to change_planner with an error
message — this is the cycle. After 3 failed retries, marks the change as requiring human review and skips
it.
Graph edges (flow)
intent_parser →
graph_retriever
Always
intent_parser →
semantic_search
Always (runs in parallel with graph_retriever)
graph_retriever +
semantic_search →
context_builder
After both complete
context_builder →
change_planner
Always
change_planner → validator
Always
validator → pr_generator
If all checks pass
validator → change_planner
If checks fail AND retry_count < 3
■ Expose the entire engine as a FastAPI service on an internal port. The Node.js backend calls it over HTTP when
processing a Bull job — it never exposes this service externally.

Layer 6 — Change Planner & PR Generator
PR GEN
Once the LangGraph engine produces a validated change set, the PR generator takes over. This is purely
mechanical GitHub API work — no LLM calls.
PR creation sequence (5 steps via Octokit)
Step 1 — Get HEAD SHA
GET /repos/{owner}/{repo}/git/refs/heads/main
Store the SHA — this is the base for your new branch.
Step 2 — Create feature branch
POST /repos/{owner}/{repo}/git/refs
body: { ref: 'refs/heads/thinksync/{intent-slug}', sha: {head_sha} }
The slug is a URL-safe version of the intent string plus a timestamp to prevent collisions.
Step 3 — Commit all changed files in one call
Do NOT commit files one at a time in a loop — this burns API calls and can hit rate limits. Use the Git
Trees API to batch everything into a single commit:
POST /repos/{owner}/{repo}/git/trees → create tree with all file blobs
POST /repos/{owner}/{repo}/git/commits → create commit pointing to that tree
PATCH /repos/{owner}/{repo}/git/refs/heads/{branch} → advance branch to new commit
Step 4 — Open the pull request
POST /repos/{owner}/{repo}/pulls
The PR body is a structured Markdown document containing:
 Summary of the intent in plain English
 Table of all changed files with per-file justifications
 Dependency impact report — which nodes were traversed and at what graph distance
 Estimated risk level (low / medium / high) from the intent_parser output
 Rollback plan — the branch can be deleted; the base branch is untouched
Step 5 — Add labels and reviewers
POST /repos/{owner}/{repo}/issues/{pr_number}/labels
body: ['ai-generated', 'needs-review']
After PR creation
Store the PR number, URL, branch name, head SHA, and creation timestamp in PostgreSQL linked to the
job record. This is what the feedback loop reads when the PR is merged or closed.
■ Rate limit note: GitHub App installation tokens get 15,000 req/hour. The Git Trees API collapses N file commits
into 3 API calls regardless of file count — always use it for PRs with more than 1 changed file.

Layer 7 — GraphQL API, Job Queue & Webhooks
API
GraphQL API (Apollo Server on Node.js)
This is the only external-facing interface. The frontend talks exclusively to this. Define:
Queries
repository(id)
Returns repo metadata, indexing status, and graph stats
job(id)
Returns job type, status, progress percentage, and results
pullRequest(id)
Returns PR metadata, URL, and the full impact report
Mutations
connectRepository(githubRe
poUrl)
Validates access, stores repo record, enqueues ingest-repository Bull
job, returns job ID
analyzeIntent(repoId,
intentText)
Validates repo is indexed, enqueues analyze-intent Bull job, returns job
ID
Subscriptions
jobProgress(jobId)
Streams real-time updates: files_scanned, dependencies_found,
nodes_written, pr_url. This drives the live progress UI in the frontend.
■ Implement subscriptions using Redis Pub/Sub: Bull workers publish progress events to a Redis channel; the
GraphQL subscription resolver forwards them to the client over WebSocket.
Job queue — Bull + Redis
All long-running work goes through the queue. Nothing blocking runs inside a GraphQL resolver —
resolvers only create jobs and return IDs.
Job types
ingest-repository
Clones the repo, runs the AST parser, writes to Neo4j and ChromaDB.
Reports progress as percentage of files parsed.
analyze-intent
Calls the Python LangGraph engine via HTTP, then calls the PR
generator. Reports progress as named stages: parsing intent →
querying graph → building context → generating changes → validating
→ creating PR.
Worker configuration
 Run ingest workers and analyze workers as separate processes — they have different memory
profiles
 Ingest workers can run in parallel (multiple repos at once); configure concurrency per worker
 Set job timeout to 10 minutes for ingest, 5 minutes for analyze
 On failure, retry up to 2 times with exponential backoff; then mark job as failed and notify
Webhook handler


---PAGE---

An Express route that receives GitHub App events. Register the endpoint in the GitHub App configuration.
Handle these events:
pull_request (closed/merged)
Record the PR outcome in the jobs table. Feed the result (merged /
rejected / reverted) back as a reinforcement signal for future change
planning.
check_run (completed)
Record CI/CD test pass/fail on the PR. Store as a change_confidence
score on the affected function nodes in Neo4j.
push (to main)
Trigger incremental re-indexing — only re-parse files that appear in the
push's changed files list, not the full repo. Update Neo4j nodes in-place
using MERGE.
installation (created)
Store the installation ID and begin the initial repo indexing job.

Hackathon MVP — Build Order & Scope Cuts
If shipping in 48 hours, build in this exact order and apply the scope cuts noted. Every item is a working,
testable unit before moving to the next.
1. GitHub OAuth (simplified)
Plain OAuth app instead of GitHub App. Skip webhook registration.
Just get a token and clone a repo.
2. tree-sitter parser (Python
only)
Extract functions, classes, imports, calls from .py files only. Two-pass
import resolution.
3. Neo4j ingestion (minimal
schema)
File, Function, Class nodes. CALLS, IMPORTS, DEFINES edges. Skip
INHERITS and Package nodes.
4. ChromaDB + OpenAI
embeddings
Function-level chunks only. Single collection per repo. No hybrid
merging yet.
5. LangGraph (3 nodes only)
intent_parser → change_planner → pr_generator. Skip the validator
cycle and semantic_search node.
6. PR generator (simple path)
File-by-file Contents API commits (not Trees API). Single file changes
only for demo.
7. REST API (skip GraphQL)
Three endpoints: POST /connect, POST /analyze, GET /job/:id. Poll
instead of subscriptions.
That is a working end-to-end demo: connect a Python repo, type an intent, get a real GitHub PR with an
impact report. Everything else — multi-language, validator cycle, Trees API batching, webhook feedback,
GraphQL subscriptions, job queue — is Phase 2.
